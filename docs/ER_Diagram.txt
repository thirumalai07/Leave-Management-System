## Relationships Summary:

1. **EMPLOYEES --(1:N)--> EMPLOYEES(Self referencing)**
   - employee → manager

2. **EMPLOYEES --(1:N)--> LEAVE_REQUESTS**
   - One employee can have many leave requests

3. **EMPLOYEES --(1:N)--> LEAVE_BALANCES**
   - One employee can have leave balances for different leave types and years

4. **EMPLOYEES --(1:N)--> NOTIFICATIONS**
   - One employee can have many notifications

5. **EMPLOYEES --(1:N)--> AUDIT_LOGS**
   - One employee can have many audit log entries

6. **LEAVE_REQUESTS --(M:1)--> EMPLOYEES**
   - Many leave requests can be approved by one employee (approver)

## Key Constraints:

- `email` in EMPLOYEES table is UNIQUE
- `date` in HOLIDAYS table is UNIQUE
- `name` in DEPARTMENTS table is UNIQUE
- Composite UNIQUE constraint on (employee_id, leave_type, year) in LEAVE_BALANCES
- Foreign key relationships with CASCADE DELETE for data integrity
- Self-referencing foreign key for manager hierarchy
